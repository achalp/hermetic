# CSV Insight — Application Specification

## 1. What This Is

A self-contained web application where a user uploads a CSV file, asks natural language questions about their data, and receives answers rendered as a composed UI built from a fixed set of visual components. All data analysis code is generated by an LLM and executed in E2B sandboxes. The frontend never runs arbitrary code — it only renders a declarative JSON component tree returned by the server.

## 3. Component Registry (The Complete Set)

The LLM may only compose UIs using the following component types. The frontend renderer has a 1:1 mapping from each `type` string to a React component. No other types are valid.

### 3.1 Layout Components

| Type | Purpose | Props |
|------|---------|-------|
| `layout-row` | Horizontal flex container | `gap?: number`, `align?: "start" \| "center" \| "end" \| "stretch"` |
| `layout-column` | Vertical flex container | `gap?: number` |
| `layout-grid` | CSS grid for card arrangements | `columns: number`, `gap?: number` |

### 3.2 Content Components

#### `stat-card`
A single KPI or metric with optional trend indicator.
```json
{
  "type": "stat-card",
  "props": {
    "label": "Average Revenue",
    "value": "$142,300",
    "change": "+12.4%",
    "trend": "up",
    "description": "vs. previous quarter"
  }
}
```
| Prop | Type | Required | Notes |
|------|------|----------|-------|
| `label` | string | yes | Metric name |
| `value` | string | yes | Formatted display value |
| `change` | string | no | e.g. "+12.4%", "-3.2%" |
| `trend` | `"up" \| "down" \| "flat"` | no | Drives color/icon |
| `description` | string | no | Contextual subtitle |

#### `text-block`
Narrative text — the LLM's explanation, insight, or annotation.
```json
{
  "type": "text-block",
  "props": {
    "content": "Revenue peaked in Q3, driven primarily by the Enterprise segment...",
    "variant": "insight"
  }
}
```
| Prop | Type | Required | Notes |
|------|------|----------|-------|
| `content` | string | yes | Markdown-supported text |
| `variant` | `"body" \| "insight" \| "warning" \| "heading"` | no | Styling variant. Default `"body"` |

#### `data-table`
A tabular view of data, paginated client-side.
```json
{
  "type": "data-table",
  "props": {
    "columns": ["Region", "Q1", "Q2", "Q3", "Q4"],
    "rows": [
      ["North", "$120K", "$135K", "$142K", "$128K"],
      ["South", "$98K", "$102K", "$115K", "$108K"]

    "caption": "Quarterly revenue by region",
    "highlight_max": true

```
| Prop | Type | Required | Notes |
|------|------|----------|-------|
| `columns` | string[] | yes | Column headers |
| `rows` | string[][] | yes | Row data (pre-formatted strings) |
| `caption` | string | no | Table title |
| `highlight_max` | boolean | no | Highlight highest value per row |
| `highlight_min` | boolean | no | Highlight lowest value per row |
| `max_rows` | number | no | Client-side pagination threshold. Default 20 |

## 4. UI JSON Schema

Every server response contains a single `UIResponse` object:

```typescript
interface UIResponse {
  /** Unique response ID */
  id: string;

  /** The component tree to render */
  components: ComponentNode[];

  /** Raw data used (for debug/export, not rendered) */
  _debug?: {
    generated_code: string;
    execution_ms: number;
    sandbox_id: string;
  };
}

interface ComponentNode {
  /** Must be a registered component type from Section 3 */
  type: string;

  /** Props specific to the component type */
  props: Record<string, any>;

  /** Nested components (only valid for layout types) */
  children?: ComponentNode[];
}
```

**Validation rules:**
- `children` is only allowed on `layout-row`, `layout-column`, and `layout-grid`.
- All other components are leaf nodes.
- Maximum nesting depth: 3 levels.
- Maximum total components per response: 20.

## 5. Backend Pipeline (Step by Step)

### Step 1: CSV Ingestion

On upload, the server:
1. Saves CSV to temp storage and assigns a `csv_id`.
2. Parses headers, dtypes, row count, and a 5-row sample.
3. Stores this **schema snapshot** for prompt construction.

```typescript
interface CSVSchema {
  csv_id: string;
  filename: string;
  row_count: number;
  columns: {
    name: string;
    dtype: "string" | "number" | "date" | "boolean";
    sample_values: string[];   // first 5 non-null values
    null_count: number;
  }[];
  sample_rows: Record<string, string>[];  // first 5 rows
}
```

### Step 2: LLM Call — Code Generation

**System prompt structure:**

```
You are a data analyst. You will be given a CSV schema and a user question.

Your job is to write a single Python script that:
1. Reads the CSV from "/data/input.csv"
2. Performs the necessary analysis using pandas, numpy, scipy
3. Prints a single JSON object to stdout with this exact structure:
   {
     "results": { ... },       // computed values, aggregations, statistics
     "chart_data": [ ... ],    // arrays formatted for chart components
     "images": { "key": "base64..." }  // any matplotlib/seaborn output
   }

Rules:
- Use pandas for all data manipulation.
- For charts that Recharts can handle (bar, line, area, pie, scatter), return
  the data as JSON arrays under chart_data. Do NOT generate matplotlib for these.
- Use matplotlib/seaborn ONLY for: heatmaps, correlation matrices, distribution
  plots with KDE, or other complex visualizations. Save as base64 PNG.
- Always handle missing values gracefully.
- Never use print() for anything except the final JSON output.
- Do not install packages. Available: pandas, numpy, scipy, matplotlib, seaborn, scikit-learn.
```

**User prompt:**

```
## CSV Schema
Filename: {filename}
Rows: {row_count}
Columns:
{for each column: "- {name} ({dtype}) — sample: {sample_values}"}

## Sample Rows
{JSON of first 5 rows}

## Question
{user_question}
```

**LLM returns:** A fenced Python code block.

### Step 3: E2B Execution

```python
from e2b_code_interpreter import Sandbox

def execute_analysis(csv_bytes: bytes, code: str) -> dict:
    with Sandbox.create(template="csv-insight") as sbx:
        # Upload the CSV into the sandbox
        sbx.files.write("/data/input.csv", csv_bytes)

        # Execute the LLM-generated code
        execution = sbx.run_code(code)

        if execution.error:
            return { "error": execution.error.traceback }

        # Parse the JSON output from stdout
        import json
        result = json.loads(execution.text)

        # Collect any matplotlib images from rich results
        for r in execution.results:
            if r.png:
                result.setdefault("images", {})
                result["images"][f"plot_{len(result['images'])}"] = r.png

        return result
```

**E2B sandbox template** (`csv-insight`) is pre-built with:
- Python 3.11
- pandas, numpy, scipy, matplotlib, seaborn, scikit-learn
- No network access (sandboxed)

### Step 4: LLM Call — UI Composition

A second LLM call takes the execution results and composes the component tree.

**System prompt:**

```
You are a UI composer. Given analysis results and a user question, produce a
JSON response that arranges findings into a component tree.

Available components: stat-card, text-block, data-table, bar-chart, line-chart,
area-chart, pie-chart, scatter-chart, annotation, trend-indicator, chart-image,
layout-row, layout-column, layout-grid.

Rules:
- Start with a text-block (variant: "heading") summarizing the answer.
- Use stat-cards for key metrics. Group them in a layout-grid (columns: 2-4).
- Use the appropriate chart type for the data shape.
- Add annotations for outliers, notable patterns, or caveats.
- Use trend-indicators when comparing two time periods.
- Use chart-image ONLY when an image was generated in the sandbox.
- Keep total component count under 20.
- Return ONLY valid JSON matching the UIResponse schema. No markdown fencing.
```

**User prompt:**

```
## Original Question
{user_question}

## Analysis Results
{JSON of execution results}

## Available Images
{list of base64 image keys, if any}
```

**LLM returns:** A `UIResponse` JSON object.

### Step 5: Validation & Streaming

Before sending to the client:
1. Validate every `type` against the component registry. Reject unknowns.
2. Validate `children` only on layout components.
3. Enforce max depth (3) and max component count (20).
4. Stream the response as server-sent events (SSE), sending components as they are validated.

## 6. API Contract

### `POST /api/upload`

Upload a CSV file.

**Request:** `multipart/form-data` with file field `csv`
**Response:**
```json
{
  "csv_id": "abc123",
  "schema": { /* CSVSchema object */ }
}
```

### `POST /api/query`

Ask a question about an uploaded CSV.

**Request:**
```json
{
  "csv_id": "abc123",
  "question": "What are the top 5 products by revenue and how has their trend changed?"
}
```

**Response:** SSE stream of `UIResponse` (streamed as the composition LLM generates it).

Event types:
```
event: status
data: {"stage": "generating_code"}

event: status
data: {"stage": "executing", "sandbox_id": "sbx_xxx"}

event: status
data: {"stage": "composing_ui"}

event: component
data: {"type": "text-block", "props": {...}}

event: component
data: {"type": "layout-grid", "props": {...}, "children": [...]}

event: done
data: {"id": "resp_xxx", "total_components": 8, "execution_ms": 2340}
```

## 7. Frontend Renderer

### Core Renderer

```tsx
// componentRegistry.ts
import { StatCard } from "./components/StatCard";
import { BarChart } from "./components/BarChart";
// ... all component imports

export const registry: Record<string, React.ComponentType<any>> = {
  "stat-card": StatCard,
  "text-block": TextBlock,
  "data-table": DataTable,
  "bar-chart": BarChartComponent,
  "line-chart": LineChartComponent,
  "area-chart": AreaChartComponent,
  "pie-chart": PieChartComponent,
  "scatter-chart": ScatterChartComponent,
  "annotation": Annotation,
  "trend-indicator": TrendIndicator,
  "chart-image": ChartImage,
  "layout-row": LayoutRow,
  "layout-column": LayoutColumn,
  "layout-grid": LayoutGrid,
};

// Renderer.tsx
function RenderNode({ node }: { node: ComponentNode }) {
  const Component = registry[node.type];
  if (!Component) return null; // unknown type — skip silently

  if (node.children) {
    return (
      <Component {...node.props}>
        {node.children.map((child, i) => (
          <RenderNode key={i} node={child} />
        ))}
      </Component>
    );
  }

  return <Component {...node.props} />;
}
```

### Progressive Rendering

As SSE `component` events arrive, append them to a state array. The UI builds up incrementally — stat cards appear first, then charts fill in. Use skeleton placeholders during the `generating_code` and `executing` stages.

## 8. Error Handling

| Failure | Recovery |
|---------|----------|
| CSV parse error | Return error to user at upload time. Reject non-CSV. |
| LLM generates invalid Python | E2B returns traceback. Retry once with the error appended to the prompt: "Your code failed with: {error}. Fix it." |
| E2B execution timeout (>30s) | Kill sandbox. Return annotation component with error. |
| E2B sandbox creation fails | Fallback: return text-only response from LLM without code execution. |
| LLM returns invalid UI JSON | Validate and strip invalid components. Render what's valid. |
| Component type not in registry | Skip silently. Log for monitoring. |

**Retry policy:** One automatic retry on code execution failure. On second failure, return the error as an `annotation` component with `severity: "error"`.

## 11. Security Constraints

1. **No code runs on the server.** All LLM-generated code executes only inside E2B sandboxes.
2. **E2B sandboxes have no network access.** Data stays in the sandbox.
3. **CSV size limit:** 50MB max upload. Reject larger files at the API layer.
4. **Sandbox timeout:** 30 seconds max execution time.
5. **No persistent state in sandboxes.** Each query gets a fresh sandbox.
6. **UI JSON is validated** against the component registry before rendering. Unknown types are dropped.
7. **CSV data never sent to the LLM.** Only the schema and 5 sample rows are included in prompts. The full CSV is only uploaded to the E2B sandbox.

## 12. Example End-to-End Flow

**User uploads:** `sales_2024.csv` (columns: date, region, product, revenue, units, cost)

**User asks:** "Which region is most profitable, and show me the monthly trend?"

**Step 1 — Code generation LLM produces:**
```python
import pandas as pd
import json

df = pd.read_csv("/data/input.csv", parse_dates=["date"])
df["profit"] = df["revenue"] - df["cost"]
df["month"] = df["date"].dt.to_period("M").astype(str)

# Region profitability
region_profit = df.groupby("region")["profit"].sum().sort_values(ascending=False)
top_region = region_profit.index[0]
top_profit = region_profit.iloc[0]

# Monthly trend by region
monthly = df.groupby(["month", "region"])["profit"].sum().reset_index()
pivot = monthly.pivot(index="month", columns="region", values="profit").fillna(0)

print(json.dumps({
    "results": {
        "top_region": top_region,
        "top_profit": float(top_profit),
        "total_profit": float(df["profit"].sum()),
        "region_summary": region_profit.to_dict(),
        "margin": float(df["profit"].sum() / df["revenue"].sum() * 100)
    },
    "chart_data": {
        "monthly_trend": [
            {"month": idx, **{col: float(row[col]) for col in pivot.columns}}
            for idx, row in pivot.iterrows()
        ],
        "region_bars": [
            {"region": k, "profit": float(v)}
            for k, v in region_profit.items()
        ]
    }
}))
```
